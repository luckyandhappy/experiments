### Ⅰ	论文目的概述

​		抓住模型训练、评估场景中存在的两个重复性并进行针对性优化：

1. 数据集重复性：针对**同一数据集可能会被多次执行**、**数据集内数据提前可见**这一见解，设计高效的前缀缓存计算、填充机制以及请求调度机制
2. 模型重复性：针对同一**模型可能会在多个数据集上执行**、**模型内安全性表征存在固定性**这一见解，设计高效的表征分类器训练、推理流水线（包括表征收集器、表征训练器、表征分类器）（需要在模型底层进行优化，最好将其嵌入计算图，需要优化流水线气泡）

### Ⅱ	基于 `CSTrie` 的前缀计算与调度策略

​		基于模型训练、评估场景中数据集完全已知、会被多次复用这一先验知识，本论文提出使用 `CSTrie` (`Cache Scheduler Trie`) 对数据集中所有请求的前缀 Token 进行预先计算，并基于计算结果将缓存填充入模型执行环境，避免现有的针对服务场景优化的模型执行框架 "尽可能多缓存" 策略带来的不必要缓存开销。基于 `CSTrie` 的前缀计算与调度策略，基本过程如下：

1. 首次执行时，加载整个数据集 (或特定大小的数据分片)，使用纵向构建算法构建 `CSTrie`
2. 基于已有的 `CSTrie`，使用启发式调度算法将其中的 "根 - 叶子" 路径打包为模型执行的批，最小化批内请求的相同前缀，最大化批间前缀复用
3. 首次执行完成后，将模型内与 `CSTrie` 对应的 `KV` 缓存进行保留，下次执行时可直接加载到显存中

##### 2.1	`CSTrie` 的数据结构

​		当前阶段，`CSTrie` 的节点设计为：

```
{
	token: 节点所代表的Token
	depth: 节点所在深度(即节点对应的Token在请求中的index)
	request_ids: 共享公共前缀为从根节点到本节点的所有请求的ID集合(不包括共享前缀还能进一步扩展的那些请求)
	children: 子节点字典, key为下一子节点的Token
}
```

##### 2.2	`CSTrie` 的纵向构建

​		考虑到数据集中的样本可共享前缀的长度一般远小于其整体长度，因此遍历逐个请求序列、先构建再减枝的横向构建策略会遍历大量不必要的序列，带来显著的开销。因此 `CSTrie` 采用纵向构建方式进行构建

​		对于数据集 A 中的四个请求序列：`1:[1,2,3,4,5], 2:[1,2,4,5,6], 3:[1,3,5,7,9], 4:[2,3,4,5,6]`，纵向构建方式的步骤为：

1. 根节点的 `request_ids = {0,1,2,3}` (四个请求).  发现 Token 1 被请求 0, 1, 2 共享 → 在当前节点下创建代表 Toke 1 的子节点, 并将请求 {0, 1, 2} 记录移动到该节点中; 请求 3 留在此节点
2. 进入子节点 (`Token=1`), `request_ids = {0,1,2}`.  发现 Token 2 被请求 0, 1 共享 → 在当前节点下创建代表 Token 2 的子节点, 并将请求 {0, 1} 记录移动到该节点中; 请求 2 留在此节点
3. 进入子节点 (`Token=2`), `request_ids = {0,1}`. 发现请求 0, 1 无更多共享 Token → 停留在此节点

这四个请求构建出的 `CSTrie` 打印出来如下所示：

```
[ROOT depth=0 reqs=[('A', 3)]]
  [token=1 depth=1 reqs=[('A', 2)]]
    [token=2 depth=2 reqs=[('A', 0), ('A', 1)]]
```

##### 2.3	`CSTrie` 的合并

​		当同时对多个数据集进行评估时，需要对代表每个数据集的 `CSTrie` 进行合并。由于 `CSTrie` 的前缀构建方式已足够优化，因此在数据集规模不大的情况下，可以直接基于两个数据集中的所有样本重新构建一个 `CSTrie`.  若两个 `CSTrie` 对应的数据集规模过大，则采用有限合并策略：

1. 先合并 `CSTrie` 中的所有分支，统计请求数量发生改变的节点
2. 从每个请求数量发生改变的节点出发，进行分支延展操作，即加载分支对应的少量请求，并从当前位置开始再次进行前缀计算

有限合并策略本质上是用时间换空间。假设存在数据集 A：`1:[1,2,3,4,5], 2:[1,2,4,5,6], 3:[1,3,5,7,9], 4:[2,3,4,5,6]` 和数据集 B：`1:[1,2,3,4,8], 2:[1,2,4,5,9], 3:[1,3,5,7,10], 4:[2,3,4,5,11]`，分别构建的 `CSTrie` 如下：

```
[ROOT depth=0 reqs=[('A', 3)]]
  [token=1 depth=1 reqs=[('A', 2)]]
    [token=2 depth=2 reqs=[('A', 0), ('A', 1)]]
```

```
[ROOT depth=0 reqs=[('B', 3)]]
  [token=1 depth=1 reqs=[('B', 2)]]
    [token=2 depth=2 reqs=[('B', 0), ('B', 1)]]
```

合并后的 `CSTrie` 为：

```
[ROOT depth=0 ]
  [token=1 depth=1 ]
    [token=2 depth=2 ]
      [token=3 depth=3 reqs=[('A', 0), ('B', 0)]]
      [token=4 depth=3 reqs=[('A', 1), ('B', 1)]]
    [token=3 depth=2 ]
      [token=5 depth=3 ]
        [token=7 depth=4 reqs=[('A', 2), ('B', 2)]]
  [token=2 depth=1 ]
    [token=3 depth=2 ]
      [token=4 depth=3 ]
        [token=5 depth=4 reqs=[('A', 3), ('B', 3)]]
```

##### 2.4	`CSTrie` 的分裂

​		在分布式场景下，需要根据不同算力资源的能力对 `CSTrie` 及其对应的数据集进行切分。`CSTrie` 的切分遵循纵向原则，假设存在数据集 A 对应的 `CSTrie` 和数据集 B 对应的 `CSTrie`，在纵向原则下切分为 `A1`，`A2`，`B1`，`B2` 四片后，`A1` 和 `B1`、`A2` 和 `B2` 分别合并能够有更大几率存在可共享前缀。例如，2.3 中合并后的 `CSTrie`，按 0.375 的比例进行分裂后，得到的两棵子树分别为：

```
[ROOT depth=0 ]
  [token=1 depth=1 ]
    [token=2 depth=2 ]
      [token=3 depth=3 reqs=[('A', 0), ('B', 0)]]
      [token=4 depth=3 reqs=[('A', 1)]]
```

```
[ROOT depth=0 ]
  [token=1 depth=1 ]
    [token=2 depth=2 ]
      [token=3 depth=3 reqs=[('B', 0)]]
    [token=3 depth=2 ]
      [token=5 depth=3 ]
        [token=7 depth=4 reqs=[('A', 2), ('B', 2)]]
  [token=2 depth=1 ]
    [token=3 depth=2 ]
      [token=4 depth=3 ]
        [token=5 depth=4 reqs=[('A', 3), ('B', 3)]]
```

> 备注：这一分裂方式可能不是最优

##### 2.5	基于 `CSTrie` 的启发式填充调度算法

​		基于 `CSTrie` 的前缀缓存填充时，存在以下问题：同一批次执行的请求，如 `[1,2,3,4,5]` 和 `[1,2,3,4,6]`，即使存在较长的共享前缀，也无法实现缓存共享。对于流式异步动态批处理的情况，存在较长共享前缀的请求在调度顺序上离得越近，越容易被框架打包到一个批中执行。因此提出了一种启发式的填充调度算法，在调度时尽量将共享前缀较多的分支分散开，使不同分支的请求被放入一个批中执行，以最大化填充时的前缀缓存命中。该算法分两个阶段：

1. 每个 "根-叶子" 路径中取出第一条请求执行 (进行缓存预填充)
2. 收集所有未在第一阶段执行的请求，依次执行

算法在第一阶段使用一个辅助队列来从根节点开始的分支裂解。当这个辅助队列中分支数大于批大小时，从每个分支中获取按 `DFS` 顺序的第一个叶子路径，将其第一个请求放入批；同时将路径上其他分支全部加入辅助队列，源码如下所示：

```
def split_branch(node: XXXTrieNode) -> RequestID:
    """DFS 找出分支的第一个叶节点, 并在沿途进行分支裂解"""
    nonlocal scheduler_queue, wait_requests
    if len(node.children) == 0:
        # 叶子节点, 将第一个请求返回, 将其他请求放入等待集合
        wait_requests.extend(islice(node.request_ids, 1, None))
        return next(iter(node.request_ids))
    else:
        # 将其他分支对应的节点加入调度队列后, 进入第一个分支
        for value in islice(node.children.values(), 1, None):
            scheduler_queue.append(value)
        return split_branch(next(iter(node.children.values())))
```

当辅助队列中分支数小于批大小时，先尝试对已有分支进行进一步主动裂解，裂解函数源代码如下：

```
def split_child(node: XXXTrie):
    """从根向叶的主动裂解"""
    nonlocal scheduler_queue
    if len(node.children) >= 2:
        for child in node.children.values():
            scheduler_queue.append(child)
    elif len(node.children) == 1:
        split_child(list(node.children.values())[0])
    else:
        scheduler_queue.append(node)
```

裂解无效说明辅助队列中已经全是叶子节点。此时直接进入第二阶段

### Ⅲ	基于表征的样本分类器

表征分类器原理参考已有论文，构建一个基于 Hidden State 的线性分类探针

##### 3.1	计算样本均值方差

​		实验假设所有的对抗样本和良性样本的 Hidden State 激活具有一致性。在 Prompt 预填充的最后一个 Token 位置上，设 Token 化后的输入序列长度为 $T$，模型的 Decoder 拥有 $L$ 层，隐藏状态大小为 $D$.  则对于每个输入样本 $s$，首先提取其 Prompt 在预填充后的最后一个有效 Token 位置的 24 层隐藏状态：
$$
\text{hidden\_states}=(H^0,\ H^1,\ ...,\ H^l,\ ...,\  H^{L-1})
$$
其中每个 Decoder 层对应的表征 $H^l \in \mathbb{R}^D$.  在得到所有输入样本的隐藏状态后，计算每层的表征的均值 $\mu_l$ 和标准差 $\sigma_l$：
$$
\mu_l=\frac{1}{N}\sum_{i=1}^NH_i^l,\ \ \ \ \ \ \ \ \sigma_l=\sqrt{\frac{1}{N}\sum_{i=1}^N(H_i^l-\mu_l)^2}
$$
为节省存储开销并嵌入计算图，这一步采用的是流式计算

##### 3.2	训练分类器

​		在计算出统计量后，对后续样本的所有表征进行标准化：
$$
\hat{H}^l=\frac{H^l-\mu_l}{\sigma_l}
$$
并使用 Logistic Regression 分类器，对于第 $l$ 层，将判别函数设置为
$$
z_l=w_l^T\hat{H}^l+b_l
$$
样本最终预测为安全的置信度
$$
P=\text{Sigmoid}(z_l)=\frac{1}{1+\text{exp}(-z_l)}
$$
默认设置 $P \geq 0.5$ 样本为安全样本

> 当前的分类器训练过程仍采用样本收集 → 传到 CPU → 训练 → 回到 GPU 加载这种原始实现，似乎应采用更高效的方法

##### 3.3	分类器使用

​		在最后一个 Token 位置的所有层分类器训练完成后，选择分类效果最好的一个进行加载，对请求预填充后的对应 Token 及层位置上的表征进行分类，若分类结果为对抗性，则跳过请求的解码过程。在 500 对抗性和 500 良性请求的数据集上，采用这一策略可节省 45% 的评估时间
